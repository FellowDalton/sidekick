# Sidekick — `raw/` → wiki: the orchestrator's reusable memory

**Date:** 2026-06-19
**Status:** Approved (design), pending implementation plan
**Scope:** Add the "Layer 2" knowledge loop the README defers: the orchestrator (Claude Code) accumulates durable life-knowledge so it stops re-researching the same facts. Implemented as two vault folders (`raw/`, `wiki/`) browsed in Obsidian, a CLAUDE.md protocol for the loop, and one deterministic helper (`sidekick wiki`) that regenerates a map-of-content index. No HTML view, no new dependency, no change to the ledger/dashboard data path.

---

## 1. Why

The orchestrator researches the same recurring subjects repeatedly (the car insurer, the dentist, the landlord) with no memory between sessions. The README reserves the "LLM-wiki pattern" for exactly this — "the orchestrator's fuzzy life-knowledge (`raw/`, deferred), never the integrity-critical ledger" — but it was never built and there is no `raw/` directory.

The vault is already an Obsidian vault of markdown (`tasks/*.md`), so the wiki needs no generated HTML view: **Obsidian is the browse surface** (backlinks, `[[wikilinks]]`, graph, search). Since the primary purpose is **orchestrator reuse** (not a human-facing reference), the orchestrator reads/writes the markdown directly, exactly as it already does with task files.

The design's one tension is the project's deterministic-spine rule: code writes data, the model only drops finished artifacts. The wiki is the *sanctioned exception* — it is allowed to be model-maintained and lossy. We preserve the rule by keeping the only code (`sidekick wiki`) purely mechanical (it indexes notes; it never synthesizes), and by keeping the wiki entirely outside the ledger/task data path.

## 2. Goals / non-goals

**Goals**
- The orchestrator has a place to accumulate durable facts (`wiki/`) and a habit (CLAUDE.md protocol) of reading it *before* researching and updating it *after*.
- Verbatim research prose is archived (`raw/`) as the lossless source behind each lossy wiki note.
- A stable entry point (`wiki/_index.md`) exists for both Obsidian and the orchestrator, regenerated deterministically by `sidekick wiki`.
- The whole thing is browsable in Obsidian with zero custom UI.

**Non-goals (YAGNI)**
- No generated HTML wiki view (Obsidian is the UI). Approach C rejected.
- No model logic in `sidekick.py` — `sidekick wiki` is deterministic indexing only.
- No automatic raw→wiki synthesis by code — folding prose into topic notes is the orchestrator's judgment (the protocol).
- No change to `ledger.jsonl`, `tasks/*.md`, the dashboard feed, or the nudge.
- No new runtime dependency (still `pyyaml`).
- No staleness/auto-expiry machinery — the `updated:` date lets the orchestrator judge staleness itself.

## 3. Architecture

Two new vault folders + a protocol + one deterministic helper. The model maintains the knowledge; code only indexes it.

```
sidekick/                      (vault root)
├── raw/                       NEW — verbatim research dumps (archival)
│   └── YYYYMMDD-<slug>.md
├── wiki/                      NEW — synthesized topic notes (the memory)
│   ├── <topic>.md             one per recurring subject
│   └── _index.md              GENERATED by `sidekick wiki` (map-of-content)
├── sidekick.py               EDITED — add the deterministic `wiki` command
├── sidekick                  EDITED — wrapper routes `sidekick wiki`
├── tests/test_wiki.py        NEW — stdlib unittest for the index build
├── CLAUDE.md                 EDITED — the loop protocol + routine row
└── README.md                 EDITED — short "Layer 2: the wiki" section
```

## 4. Components

### 4.1 `raw/` — verbatim research archive
- **One file per research session:** `raw/YYYYMMDD-<slug>.md` (`<slug>` derived from the topic/task, same `slug()` style as task ids).
- **Frontmatter:**
  ```yaml
  ---
  task: <source task id, e.g. 20260613-sort-the-car-insurance-renewal>
  topic: <wiki topic slug it feeds, e.g. car-insurance>
  created: <ISO timestamp>
  ---
  ```
- **Body:** the orchestrator's raw research prose, unedited.
- **Role:** lossless source/receipts behind the lossy wiki note. Referenced from the wiki note's `sources:`. Not parsed by any code — purely an archive.

### 4.2 `wiki/<topic>.md` — synthesized topic notes (the reusable memory)
- **One note per recurring subject**, free-form areas of life (`car-insurance`, `dentist`, `landlord`, `passport`) — **not** the task categories (those are too coarse). The orchestrator chooses/normalizes topic slugs.
- **Frontmatter:**
  ```yaml
  ---
  summary: One-line description, shown in the index.
  updated: 2026-06-19            # ISO date the orchestrator last revised this note
  sources:                       # provenance: raw dumps and/or task ids
    - "[[20260619-car-insurance]]"
    - 20260613-sort-the-car-insurance-renewal
  ---
  ```
- **Body:** durable, reusable facts in the orchestrator's words (numbers, account refs, decisions, gotchas), cross-linked with `[[wikilinks]]`. Obsidian renders backlinks/graph from these.
- **Role:** what the orchestrator reads *before* researching and updates *after*. The note IS the memory.

### 4.3 CLAUDE.md protocol — the loop (the model-driven part)
Add a section to CLAUDE.md describing the loop the orchestrator follows:
- **Before researching a task:** read `wiki/_index.md`, then grep/glob `wiki/` for the task's subject; read any matching topic note(s) and reuse known facts instead of re-researching.
- **After researching:** (a) write the verbatim prose to `raw/<date>-<slug>.md` with the `task`/`topic`/`created` frontmatter; (b) fold the durable facts into `wiki/<topic>.md` (create or update it; refresh `summary`/`updated`/`sources`; add `[[links]]`); (c) run `sidekick wiki` to refresh the index.
- **Integrity:** the wiki is fuzzy/lossy by design and lives entirely outside the ledger/task data path. Wiki work must never touch `ledger.jsonl` or task frontmatter. `wiki/_index.md` is generated — never hand-edit it.

### 4.4 `sidekick wiki` — the deterministic index helper (the only code)
- **Command:** `python sidekick.py wiki` (and `sidekick wiki` via the wrapper). Rebuilds `wiki/_index.md`.
- **Behavior:**
  1. If `wiki/` does not exist → print a notice and exit 0 (no-op); do not create it.
  2. Scan `wiki/*.md` except `_index.md`. For each note, via the existing `read_note()`:
     - `title` = first `# ` heading in the body, else the filename stem.
     - `summary` = frontmatter `summary` (str) or `""`.
     - `updated` = frontmatter `updated` (str) or `""`.
     - `stem` = filename without `.md` (the Obsidian link target).
  3. Sort entries by `updated` descending (ISO dates sort lexically; empty `updated` sorts last), then `title` ascending as a tiebreaker — **fully deterministic, no filesystem mtime**.
  4. Atomically write `wiki/_index.md` (tmp + `os.replace`, the existing pattern):
     ```markdown
     <!-- GENERATED by `sidekick wiki` — do not hand-edit. -->

     # Wiki — reusable memory

     - [[<stem>|<title>]] — <summary>  ·  updated <updated>
     - [[<stem2>|<title2>]] — <summary2>
     ...
     ```
     Omit ` — <summary>` when summary is empty; omit `  ·  updated <updated>` when updated is empty. When there are no topic notes, write the header + a single line `_No topics yet._`.
  5. Print `wrote wiki/_index.md — N topics`.
- **Determinism / no churn:** the index content is a pure function of the notes (NO generation timestamp), so re-running with unchanged notes produces a byte-identical file — no spurious git diffs (an improvement over `sidekick-data.js`).
- **Degradation:** a note whose frontmatter fails to parse (or isn't a dict) falls back to `title=stem`, empty summary/updated — it never crashes the build.
- **Boundary:** reads only `wiki/*.md`; writes only `wiki/_index.md`. Never touches `ledger.jsonl`, `tasks/`, or `sidekick-data.js`. Not called by `regenerate` (separate cadence).

## 5. Data flow

```
research (orchestrator) ─► raw/<date>-<slug>.md     (verbatim archive)
                        └► wiki/<topic>.md           (synthesized, [[linked]])
                                  │  sidekick wiki (deterministic index)
                                  ▼
                           wiki/_index.md (MOC) ─► Obsidian browses
                                  ▲
   next task: orchestrator reads _index + topic notes BEFORE researching
```

## 6. Error handling / edge cases
- **No `wiki/` dir:** `sidekick wiki` no-ops with a notice (does not create the folder). The before-research read simply finds nothing and the orchestrator seeds the first note.
- **Empty `wiki/`:** index is written with the `_No topics yet._` body.
- **Malformed/missing frontmatter in a note:** degrade to `title=stem` (per §4.4), empty summary/updated; never crash.
- **`updated` not a valid date:** treated as an opaque string for sorting/display (ISO assumed; garbage sorts wherever it falls lexically) — acceptable for a fuzzy layer; no validation.
- **`_index.md` present:** always excluded from the scan so it never indexes itself.

## 7. Testing
A stdlib `unittest` (`tests/test_wiki.py`), same shape as `test_regenerate.py` — runs the real CLI as a subprocess against a throwaway vault via `SIDEKICK_VAULT`:
- **Build + sort + format:** seed `wiki/car-insurance.md` (`updated: 2026-06-13`, summary, H1 "Car insurance") and `wiki/dentist.md` (`updated: 2026-06-16`, summary). Run `wiki`. Assert `wiki/_index.md` exists, lists both as `[[stem|Title]]` with summaries, dentist appears before car-insurance (newer `updated` first), and the "do not hand-edit" header is present.
- **Degradation:** add `wiki/broken.md` with malformed frontmatter. Assert the build still succeeds and `broken` appears (by stem) without crashing.
- **No-churn:** run `wiki` twice; assert the two `_index.md` outputs are byte-identical.
- **Empty/no-wiki:** with no `wiki/` dir, assert exit 0 and no file created; with an empty `wiki/`, assert the `_No topics yet._` body.

## 8. Integrity & boundaries
- `sidekick wiki` is deterministic and side-effect-scoped to `wiki/_index.md`. No model logic in `sidekick.py`.
- The wiki is the sanctioned LLM-maintained layer; the deterministic spine (ledger, dashboard feed, nudge) is untouched and the wiki cannot affect scoring.
- `raw/` and `wiki/` are **tracked in git** (like `tasks/`) — they are the user's accumulated knowledge, worth versioning. `wiki/_index.md` is generated but committed (it's small, deterministic, and useful in the repo).

## 9. Out of scope (parked)
A human-facing HTML wiki view; automatic synthesis by code; staleness/expiry automation; importing existing notes; cross-vault sharing. None block this work.
